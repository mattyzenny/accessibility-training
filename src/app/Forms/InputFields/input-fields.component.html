Input Fields

Input fields are important for filling out forms or submitting data. They can also be some of the most complex components when it comes to Accessiblilty. If you think about it, an input field can be as simple as an <input> and a <label>, or as complext as a required input, an error message, and an input field where only certain special characters allowed. All of this needs to be conveyed to both sighted and non-sighted users.

What makes an accessible input?

  • A visible label must be semantically associated to the input field
  • If an error message is triggered, either on blur() or on page submit, it must be announced to the AT user.
  • If a field is required it must provide that information to both visual and non-sighted users.
  • Placeholder text must not be used as the label for an input.
  • If personal data is being collected, the proper input type and input purpose (autocomplete) must be used if applicable. There are 22 input types and 50 input purposes to choose from.

There are many other things to consider, so we must make sure each newly added enhancement to the component is accessible.

Here is a list of 4 different input elements. Lets look at how they progressively get more complex, and how each newly added component is used in an accessible way.

            data-required=true.
            The * is a common way to indicate a required field.
            Thankfully this is handled from the cel-component library where data-required=true injects
            HTML's  required=true and ARIA's
            aria-hidden=true hiding the *
            from screen readers.
            That way, both sighted users visually recognize the required field and AT user only hear
            'required'.

            We have also added new attributes data-helper-label and data-max-field-length.
            The data-max-field-length attribute limits the number of characters a user can input into the
            field.
            If we add this type of limitation, we also need to communicate this to users.

            In the current design, the data-helper-label is injected into the HTML
            <label> field.
            If we remember, aria-label will overwrite whatever the visual label shows. So in this example,
            we have removed data-field-aria-label;
            so that we can read the appended <label> (<data-helper-label>)
            content.

  2. 
        This example demonstrates how we can apply both the data-field-aria-label &
        data-field-aria-describedby together.
        The aria-label attribute overwrites the label field, and aria-describedby adds supplemental content by
        referencing any element's ID.
        In this example we reference the helper-label ID with our
        data-field-aria-describedby attribute to apply both the helper-label content along
        with the HAT associated to our data-aria-field-label.

  3.

        but we are showcasing how they are accessibily used together.

            concatenate
            multiple ID's within the DOM. Here we are referencing the ID of data-helper-label &
            data-field-aria-describedby-content
            to give us a optimally robust description for our input field.
