Landmarks are essential for accessibility, aiding both navigation and comprehension for assistive technology users. They provide the same modular structure to screen readers that visual design offers to sighted users.

Required

  • If a landmark is used more than once, it must have a unique label.
  • A page must have one <main> landmark.
  • All content must be contained inside a landmark.
  • Only one <banner>, <main>, <contentinfo> landmark is allowed per page. Learn more.

Other than quickly navigating a page, screen reader users can pull up a list of landmarks. In the screen below you can see an example of this. 

    
<body>
<div role="banner">
<div role="navigation"/>
</div>
<div role="complementary"/>
<div role="main">
...
...
<div role="navigation"/>
</div>
</body>

We can notice 2 things.

  • ✅ We are effectively using multiple <nav> landmarks with unique labels to enhance navigation and accessibility.
  • ❌ We have an empty <complimentary> landmark. This is dead code a should be moved. It provides nothing to the user and may confuse others.

Provided below, we can see we have 2 <main> landmarks. This is not allowed and should be fixed. We can also see redundant navigation landmarks, as well as redundant labelling. 

            
<body>
<div role="banner">
<div role="navigation"/>
</div>
<div role="complementary"/>
<div role="main">
<div class="app-container">
...
<div role="navigation" aria-label="secondary">
<nav aria-label="side navigation"/>
</div>
<main>
...
...
</main>
</div>
</div>
</body>

Notice that empty <complimentary> region is carried across all pages in the application. This is where MFE's should discuss who will own which landmarks on the page layout, and wrap their content accordingly.

Ask yourself, if you pulled up this list of landmarks, would you know the difference between navigation and side navigation? How about between the 2 main landmarks? If we can't make sense of it, neither can a screen reader.
