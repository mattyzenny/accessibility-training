DO NOT USE NATIVE TOOLTIPS UNLESS THEY ARE CUSTOMIZED!

Lets clear the air on what a tooltip is. Historically and colloquially, a tooltip is a native browser popup that displays on hover. This is applied with the title attribute. We can already infer a few things wrong with this. A native HTML tooltip alone does not suffice in todays accessibility standards. A tooltip may be visible for individuals using a mouse, but anyone on a mobile device or using strictly keyboard navigation will never see these tooltips.

There is langauge that excuses the title attribute for its inherent inaccessibility, but for the purposes of this training, all content should be accessible to all users.

Now using title is not wrong, and sometimes it can be quite useful, it just needs to be used sparingly. For instance, using a Native HTML tooltip on an icon button or icon link is preferred. However, it is almost always preferred, if using a tooltip, to provide a tooltip that displays onHover and onFocus.

Required

  • Icons must have a sufficient color contrast of 3:1. It must not break contrast of foreground elements. WCAG 1.4.11
  • Icons and images should not include text. WCAG 1.4.5
    1. Decorative
      • Icons must have an empty alt attribute. WCAG 1.1.1
    2. Non-Decorative
      • Icons must have a text alternative. WCAG 1.1.1
      • If interactive, it must have a tabindex WCAG 2.1.1
      • Use of color cannot be the only thing to convey meaning. WCAG 1.4.1
      • Icons must be consistently identified and used across the application. WCAG 3.2.3   WCAG 3.2.4
      • Content that appears on focus needs to be dismissable without moving the mouse. WCAG 1.4.13

Best Practice

  • Do not rely on the title attribute to provide alternative text. It should only be supplimental and non-required information.
  • Do not use the title attribute with an aria-label to avoid AT redundant annoucement.
  • Provide visible text labels, if not, provide custom tooltips on ambiguous icons.

Key Differences

  • Icon
    1. Icons do not inherently have a tabindex.
    2. If decorative, icons should have an empty alt attribute.
    3. Icons should never be used as a selectable item unless it is wrapped by a interactive element.
  • Icon-Button
    1. Icon-buttons inherently have a tabindex.
    2. Icon-buttons should perform some action.
    3. Cel-icon-buttons (currently) are not fully accessible, as they descriminate against keyboard users.
    4. Icon-buttons that use only the title attribute are not a good practice.
  • Icon-Button (custom)
    1. Icon-buttons should have a tabindex.
    2. Icon-buttons should perform some action.
    3. Icon-buttons, if ambiguous, should have onHover and onFocus tooltips.

      Ambiguous Icons
    

    

      The icons below are not universally understood. It is crucial that designs are considerate of the icons we are
      using.
      Not every user is a screen reader user. If we are using a icon-button, it should be understood for all users.
    

    

      Tooltips are the hero for these types of scenarios, where onHover() and
      onFocus() stand out.
    

  

  

    

      
      
      
    

  

  

    

      

        
      

      

        

          Icon Button
          

          <cel-icon-button
           data-color=
          "primary"
           data-name=
          "about-author"
           data-tooltip-label=
          "About Author"></cel-icon-button>
          


          Icon Button (custom)
          

          <cel-icon-button
           data-color=
          "primary"
           data-icon-button-aria-label=
          "About Author"
           data-name=
          "about-author"
           data-bs-tooltip=
          "true"
           data-bs-title=
          "About Author"></cel-icon-button>
          

        

      

    

  

  

    

      Hopefully now you can see the strength of custom tooltips.
      Notice without a visible label on icons, keyboard users have no way of understanding the purpose
      of the icons and icon-buttons.
